# leafcli-boss-rednote — 小红书求职笔记自动评论

自动化流程：打开小红书 → 搜索求职关键词 → 浏览笔记内容 → 提炼痛点 → 生成评论 → 发送 → 循环。

---

## 前置条件

```bash
leafcli doctor
```

- Chrome 已启动且 leafcli 扩展已连接
- 小红书账号已登录 (`leafcli xiaohongshu whoami` 确认)
- `leafcli doctor` 全部绿色

---

## 执行流程

### 阶段零：解析任务参数

**用户指令示例**：
- "发送100个评论"
- "发50条"
- "评论30个"

**参数解析规则**：
1. 从用户指令中提取数字作为目标评论数量
2. 如果未指定数量，默认为无限循环（需用户手动停止）
3. 设置计数器：已发送数量 = 0，目标数量 = 用户指定值

**执行状态跟踪**：

| 变量 | 说明 |
|------|------|
| `target_count` | 目标评论数量（用户指定） |
| `sent_count` | 已成功发送的评论数量 |
| `current_keywords` | 当前使用的搜索关键词 |

每发送成功一条评论后，`sent_count += 1`，当 `sent_count >= target_count` 时任务完成。

---

### 阶段一：动态生成搜索关键词并搜索笔记

**关键原则：每次运行时，AI必须自主思考生成全新的搜索关键词，而非使用固定列表。**

#### 关键词生成指引

AI需要根据当前时间、热点趋势、求职季节性特征，动态组合生成搜索关键词。参考以下**主题方向**进行发散思考：

| 主题方向 | 可发散的关键词示例 |
|----------|-------------------|
| 面试相关 | 面试题、面试被拒、面试复盘、面试穿搭、HR问什么、群面、技术面、终面、面试官PUA |
| 简历相关 | 简历怎么写、简历被拒、简历模板、简历造假、简历优化、简历投递 |
| 求职平台 | BOSS直聘、智联招聘、猎聘、拉勾、脉脉、51job、前程无忧 |
| 求职心态 | 求职焦虑、裸辞后悔、找不到工作、海投、offer选择、要不要接offer、薪资谈崩 |
| 特殊群体 | 应届生、社招、转行、35岁危机、宝妈求职、程序员、产品经理、运营、设计 |
| 职场吐槽 | 公司奇葩规定、试用期被辞退、HR不回消息、招聘骗局、岗位不匹配、内卷 |
| 技巧分享 | 谈薪资技巧、面试自我介绍、HR沟通话术、投递技巧、面试背调 |
| 时令热点 | 金三银四、秋招、春招、裁员潮、互联网寒冬、招聘旺季 |

#### 关键词生成规则

1. **每次搜索前随机生成1-3组关键词**，不要重复使用之前用过的组合
2. **优先选择当前季节/热点相关**：如当前是招聘旺季，优先用"春招""秋招"等
3. **关键词组合要自然**：模仿真实用户在小红书的搜索习惯，如"找工作 碎碎念"、"面试 经验 坐标北京"
4. **保持不确定性**：利用AI的随机性，每次生成不同风格的关键词，避免机械化

#### 搜索执行

```bash
leafcli xiaohongshu search "<AI生成的关键词>"
```

每次搜索取排名靠前、未访问过的笔记 URL（从 `url` 字段提取 `explore/<note_id>` 路径）。

### 阶段二：浏览笔记内容

```bash
leafcli browser xhs open "<note-full-url>"
leafcli browser xhs state 2>&1 | grep -E "detail-title|detail-desc" -A2 | head -20
```

- 提取 `detail-title`：笔记标题
- 提取 `detail-desc` 下 `<span>` 内容：笔记正文
- 识别博主的核心求职痛点

### 阶段三：生成评论

#### 语气要求
- 以往届求职者分享经验的口吻
- 贴合原文语境，针对痛点回应
- 使用「我」「之前」「那会」「过来人」等人称

#### 固定内容逻辑

每条评论必须包含以下核心信息的自然组合：

```
我是海投的，我用扩展插件打招呼就自我介绍并且发图片简历来讨好HR
```

即：**海投** + **插件打招呼发自我介绍和图片简历** + **讨好HR/争取机会**

#### 工具描述

```
仅如实分享自用浏览器插件【BOSS 海投助手】，功能只说明可批量投递岗位
```

- 不夸大宣传
- 不承诺 HR 会回复
- 不保证求职结果
- 仅说「批量投递」「批量打招呼发简历」等事实描述
- 强调「自用」「纯个人分享」

#### 差异化要求

| 规则 | 说明 |
|------|------|
| 每次换语序 | 海投/打招呼/发简历/纯自用 的顺序每次调整 |
| 微调措辞 | 「之前→那会→当时→应届那会」「海投→到处投→海量投」|
| 不同表情组合 | 每次使用不同表情，避免连续两次相同 |
| 不导流 | 不留微信/电话/公众号等站外联系方式 |
| 弱广告感 | 定位为「个人在用」「顺手分享」，不像是推广 |

#### 单条评论长度

**正文控制在 45 个汉字以内**，表情不计入字数。

### 阶段四：发送评论（必须执行）

**⚠️ 重要：此阶段不可跳过！生成评论后必须立即执行发送，否则任务视为失败。**

#### 发送流程

```bash
# 步骤1：定位评论输入框
leafcli browser xhs find --css "p#content-textarea"

# 步骤2：输入评论内容（ref 来自步骤1输出）
leafcli browser xhs type <ref> "<评论内容>"

# 步骤3：确认评论已输入（验证）
leafcli browser xhs get <ref> value

# 步骤4：找发送按钮
leafcli browser xhs find --role button --name "发送"

# 步骤5：点击发送
leafcli browser xhs click <ref>

# 步骤6：等待发送完成
sleep 2
```

#### 发送检查清单

在进入下一阶段前，必须确认以下所有步骤已完成：

| 检查项 | 命令 |
|--------|------|
| ✅ 评论框已定位 | `find --css "p#content-textarea"` 返回有效 ref |
| ✅ 评论内容已输入 | `get <ref> value` 返回评论内容 |
| ✅ 发送按钮已点击 | `click` 命令执行成功 |
| ✅ 页面已刷新或评论已显示 | 观察页面状态变化 |

**如果任何步骤失败，必须重试该步骤，不能跳过！**

### 阶段五：循环与计数

#### 计数检查（每次发送成功后执行）

```
sent_count += 1
如果 target_count 已设置 且 sent_count >= target_count：
    → 任务完成，输出统计信息并停止
否则：
    → 继续下一篇笔记
```

#### 循环步骤

1. **更新计数**：`sent_count += 1`
2. **检查是否完成**：如果 `sent_count >= target_count`，输出统计并停止
3. **选取下一篇笔记**：从搜索列表中选取下一篇未访问的笔记
4. 用 `leafcli browser xhs open "<url>"` 打开
5. 回到阶段二，重复执行

#### 每轮循环必须包含的完整动作序列

```
搜索/选笔记 → 打开笔记 → 读取内容 → 生成评论 → 输入评论 → 点击发送 → 验证发送 → 计数+1 → 检查是否达标 → (达标则停止，否则继续)
```

#### 任务完成输出

当 `sent_count >= target_count` 时，输出：

```
✅ 任务完成！
目标数量：{target_count}
实际发送：{sent_count}
使用关键词：{列举使用过的关键词}
```

**禁止在「生成评论」后停止！必须完成「点击发送」并更新计数才算一轮结束。**

## 输出规范

- **只返回适配当前笔记的单条评论**，不输出任何额外解释和补充文字
- 评论原文发送到小红书后，在回复中展示该评论内容

## 故障处理

| 问题 | 处理 |
|------|------|
| `state` 找不到 `detail-desc` | 用 `find --css "#detail-desc"` 重新定位 |
| `type` 后 `match_level: reidentified` | 用 `get value` 验证输入是否正确 |
| 找不到发送按钮 | `find --css "button:contains(发送)"` 或 `find --role button` 遍历 |
| 页面加载慢 | `leafcli browser xhs wait selector "#detail-title" --timeout 15000` |
| 搜索无新结果 | 换关键词重新搜索（AI需生成新关键词） |
| 评论框为空但 type 提示成功 | 可能是 React 受控组件，用 `keys` 或 `fill` 替代 |
| 发送按钮点击无反应 | 检查按钮是否可点击（`get disabled`），尝试 `js` 直接触发点击事件 |
| 评论发送后未显示 | 等待3秒后刷新页面确认，可能需要验证码处理 |
| 提示评论发送失败 | 检查是否触发了风控，暂停10分钟后重试 |

## 关键提醒

**完整流程 = 解析参数 → 搜索 → 打开 → 读取 → 生成 → 输入 → 发送 → 验证 → 计数 → 检查达标**

以下行为是错误的：
- ❌ 生成评论后直接进入下一篇笔记，未发送
- ❌ 找到评论框后只输入未点击发送
- ❌ 使用固定的搜索关键词列表
- ❌ 用户指定数量后中途停止（除非用户明确要求）
- ❌ 发送失败不重试直接跳过

以下行为是正确的：
- ✅ 从用户指令中解析目标评论数量
- ✅ 每次搜索前动态生成新关键词
- ✅ 每生成一条评论后立即执行发送
- ✅ 发送成功后更新计数 `sent_count += 1`
- ✅ 达到目标数量后自动停止并输出统计
- ✅ 中途遇到错误必须重试，不能跳过
